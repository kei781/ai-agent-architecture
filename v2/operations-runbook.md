# 운영 문서
## 하이브리드 로컬 AI 인프라 운영 가이드 (Mac Mini + Mac Studio 2-노드 기준)

> 이 문서는 **실제 운영 방법**을 정의한다.  
> 철학과 의사결정 원칙은 `architecture-philosophy-hybrid-macmini.md` 를 우선 참조하고,  
> 이 문서는 그 철학을 **현재 하드웨어와 소프트웨어 환경에 맞게 운영 단위로 풀어쓴 문서**로 사용한다.

---

## 1. 운영 목표

현재 운영 목표는 다음과 같다.

- 16GB급 노드는 **상시 제어/저장/배포 노드**로 유지한다.
- 128GB급 노드는 **로컬 추론/브라우저 자동화/이미지 생성 노드**로 운용한다.
- 내부 통신은 사설 메쉬 네트워크(Tailscale 등) 기준으로 통일하되, LAN 폴백 경로를 확보한다.
- Hermes Agent를 메인 에이전트 런타임으로 사용하고, 자기 개선 루프(스킬 + 에피소딕 메모리)를 핵심 자산으로 관리한다.
- 자동화는 에이전트 중심으로 구성하고, 별도 오케스트레이션 레이어는 기본값으로 두지 않는다.
- 모델 및 도구 선택은 **특정 이름 고정**이 아니라 **역할 기반 라우팅**으로 운영한다.

## 2. 현재 환경 매핑

| 구분 | 현재 하드웨어 | 운영 역할 |
|---|---|---|
| Node A | Mac Mini M4 16GB | Control / Data Node |
| Node B | Mac Studio M4 Max 128GB | Inference / Automation Node |

### 2.1 Node A (Control / Data Node)

기본 책임:

- Hermes Agent 메인 인스턴스 실행
- 스케줄링 (Hermes 내장 Cron + launchd 폴백)
- 코드 작업 진입점 (Claude Code, MCP)
- 관계형 DB (PostgreSQL)
- 벡터 저장소
- 에이전트 학습 자산 (스킬 DB, 에피소딕 메모리) 영속화
- 메시징 게이트웨이를 통한 알림 발송
- 외부 API 호출
- 로그, 결과, 메모리 영속화
- 워치독 프로세스 운영

### 2.2 Node B (Inference / Automation Node)

기본 책임:

- 로컬 모델 런타임
- 브라우저 자동화 (Hermes 내장 Camoufox + Playwright MCP 보조)
- 대형 분석 작업
- 이미지 생성 런타임
- 수동 관제 UI

## 3. 기본 서비스 카탈로그

아래 표는 **현재 운영에 무난한 기본값**이다.  
반드시 이 제품만 써야 한다는 뜻은 아니다.

| 기능 | 기본 선택 | 대체/보완 가능 | 비고 |
|---|---|---|---|
| 에이전트 런타임 | Hermes Agent | 동등 기능 대체재 | 자기 개선 루프 + MCP 양방향 + 메시징 내장 |
| 로컬 모델 런타임 | Ollama 계열 | vLLM, llama.cpp 계열 등 | 단일 노드 운영이 단순한 런타임 우선 |
| 수동 관제 UI | Open WebUI 계열 | 자체 UI, 다른 내부 챗 UI | 사람용 관제 패널로 사용 |
| 브라우저 자동화 | Hermes 내장 Camoufox | Playwright MCP 보조 | stealth 브라우저 기본, 필요 시 Playwright 병행 |
| 코드 작업 | Claude Code + MCP | 다른 코드 에이전트, 로컬 코드 모델 | Hermes가 MCP 서버로 노출되어 직접 연동 |
| 프로세스 관리 | launchd (macOS) | systemd (Linux 전환 시) | OS 기본 서비스 관리자 사용 |
| 스케줄링 | Hermes 내장 Cron | launchd 폴백 | 자연어 기반 스케줄 설정 가능 |
| 관계형 DB | PostgreSQL 기본 | MySQL 추가 가능 | 기본 DB는 하나로 시작 |
| 벡터 저장소 | 경량 스토어로 시작 | Qdrant 등으로 승격 가능 | 데이터 규모/보안 요건에 따라 확장 |
| 이미지 생성 | ComfyUI 계열 | 다른 노드형 생성 런타임 | 상시 실행 금지, on-demand 권장 |
| 네트워크 메쉬 | Tailscale 계열 | WireGuard 계열 등 | 외부 공개 최소화, LAN 폴백 확보 |
| 외부 장문 처리 | Stable cloud model | 다른 cloud LLM | 역할 기반 보조 슬롯 겸 폴백 경로 |
| 메시징/알림 | Hermes 내장 게이트웨이 | 별도 웹훅 구현 | 14개 플랫폼 기본 지원 |

## 4. 모델 라우팅 정책

이 문서는 모델명을 고정하지 않는다.  
대신 아래 역할별 슬롯을 사용한다.

| 슬롯 | 역할 | 추천 규모/성격 | 현재 환경에서 무난한 예시 |
|---|---|---|---|
| `MODEL_FAST` | 분류, 파싱, 짧은 요약, 단순 라우팅 | 4B~8B급 instruct | Qwen 계열 8B 전후 |
| `MODEL_ORCHESTRATOR` | 일반 추론, tool calling, 멀티스텝 작업 | 20B~35B급 instruct 또는 MoE | Qwen 계열 20B~30B급 |
| `MODEL_DEEP` | 긴 문맥, 고난도 분석, 중요한 최종 검토 | 70B급 instruct 또는 동급 | Llama 계열 70B 전후 |
| `MODEL_CODE_OPTIONAL` | 로컬 코드 보조 분석 | 코드 특화 모델 | DeepSeek-Coder 계열 등 |
| `MODEL_EXTERNAL_TEXT` | 장문 처리, 대용량 문서, 로컬 우회/폴백 | stable cloud model | Gemini Flash/Pro 계열 또는 동급 |
| `MODEL_IMAGE` | 이미지 생성/편집 | 워크플로우 기반 생성 런타임 | Flux / SDXL 계열 |

### 4.1 라우팅 규칙

**1단계 — 명시적 라우터 (초기 운영)**

- 단순 분류, 텍스트 파싱, 라벨링 → `MODEL_FAST`
- 일반 오케스트레이션, tool calling, 워크플로우 판단 → `MODEL_ORCHESTRATOR`
- 긴 문서, 깊은 분석, 최종 검토 → `MODEL_DEEP`
- 저장소 수정, 리팩토링, 패치 작성 → 우선 `Claude Code`, 필요 시 `MODEL_CODE_OPTIONAL`
- 로컬 메모리/속도/비용 균형이 맞지 않는 장문 작업 → `MODEL_EXTERNAL_TEXT`
- Node B 다운 시 추론 작업 → `MODEL_EXTERNAL_TEXT` 로 자동 폴백
- 이미지 생성 요청 → `MODEL_IMAGE` 가 붙은 별도 런타임

작업 수신 시 `MODEL_FAST`로 유형을 분류하고, 분류 결과에 따라 Hermes 인스턴스에 적절한 모델을 지정하여 전달한다.

**2단계 — 스킬 기반 자체 학습 (성숙 단계)**

Hermes의 `/model` 런타임 전환 기능을 스킬 안에서 활용한다. "이 유형의 작업이면 먼저 모델을 X로 전환하고 실행하라"는 패턴을 스킬로 학습시킨다.

### 4.2 운영 원칙

- `MODEL_FAST` 와 `MODEL_ORCHESTRATOR` 는 상시 준비 상태를 허용할 수 있다.
- `MODEL_DEEP` 는 필요 시에만 로드한다.
- `MODEL_IMAGE` 런타임은 작업 후 종료를 기본값으로 한다.
- 모델명을 직접 코드에 박지 말고, 반드시 환경변수나 설정 레지스트리로 주입한다.
- 모델 호출 실패 시 Hermes 프로바이더 라우터의 폴백 체인이 자동 동작한다.

## 5. Hermes Agent 인스턴스 토폴로지

Hermes Agent는 동기식 코어 루프(인스턴스당 1 대화)이므로, 역할별로 인스턴스를 분리한다.

### 5.1 인스턴스 구성

| 인스턴스 | 배치 노드 | 상태 | 역할 |
|---|---|---|---|
| hermes-main | Node A | 상시 | 메인 오케스트레이터. 작업 수신, 라우팅, 결과 저장, 알림 |
| hermes-scheduler | Node A | 상시 | 스케줄링 전용. Cron 작업 실행, 주기적 데이터 수집 |
| hermes-analyst | Node A/B | 필요 시 | 장시간 분석, 대형 문서 처리 전용. 완료 후 종료 |

### 5.2 인스턴스 관리

- 각 인스턴스는 별도 launchd plist로 관리한다.
- 상시 인스턴스(hermes-main, hermes-scheduler)는 launchd의 `KeepAlive` 로 자동 재시작한다.
- 필요 시 인스턴스(hermes-analyst)는 hermes-main이 SSH 또는 MCP로 기동/종료한다.
- 모든 인스턴스는 동일한 스킬 DB와 에피소딕 메모리를 참조한다(Node A의 SQLite).

## 7. 네트워크 및 인터페이스 규칙

### 7.1 네트워크

- 모든 내부 통신은 Tailscale 등 **사설 메쉬 네트워크**를 기본값으로 한다.
- 포트 포워딩과 DDNS는 가능한 한 피한다.
- 공개 인터넷 노출이 필요한 경우에도 프록시, 인증, 접근 제한을 선행한다.
- **LAN 폴백**: 두 노드 간 로컬 LAN IP를 `.env`에 별도 등록하고, 메쉬 네트워크 불통 시 이 경로로 전환한다.

### 7.2 권장 인터페이스

| 연결 주체 | 연결 방식 | 설명 |
|---|---|---|
| Node A → Node B 모델 호출 | HTTP/JSON | 가장 단순한 추론 호출 경로 |
| Node A → Node B 이미지 생성 | HTTP API 또는 SSH | 런타임 기동/작업/종료 |
| Node A → 코드 작업 | Claude Code / MCP | 저장소 중심 작업. Hermes가 MCP 서버로 노출 |
| Node A → Node B 원격 제어 | Hermes SSH 터미널 백엔드 | 에이전트가 Node B를 직접 제어 |
| Node B → Node A 작업 위임 | SSH 또는 Webhook | 운영 명령 전달 |
| 외부 단말 → 내부 UI | Tailscale 경유 HTTPS/HTTP | 수동 관제 |

### 7.3 포트 정책

포트 번호는 구현에 따라 바뀔 수 있으므로 문서에서는 **역할**만 고정한다.

- Local Model Runtime Port
- Internal Web UI Port
- Vector Store Port
- Image Runtime Port
- Custom Agent API Port

실제 포트는 `.env` 또는 운영 설정 파일에 선언한다.

## 8. 설정 레지스트리 규칙

하드코딩을 피하기 위해 아래와 같은 환경변수 기반 레지스트리를 둔다.

```env
# node identity
CONTROL_NODE_HOST=
CONTROL_NODE_TAILNET_IP=
CONTROL_NODE_LAN_IP=
COMPUTE_NODE_HOST=
COMPUTE_NODE_TAILNET_IP=
COMPUTE_NODE_LAN_IP=

# model runtime
LLM_RUNTIME_BASE_URL=
MODEL_FAST=
MODEL_ORCHESTRATOR=
MODEL_DEEP=
MODEL_CODE_OPTIONAL=
MODEL_EXTERNAL_TEXT=
MODEL_IMAGE=

# hermes agent
HERMES_CONFIG_PATH=
HERMES_SKILLS_DB_PATH=
HERMES_EPISODIC_DB_PATH=
HERMES_DEFAULT_PROVIDER=
HERMES_FALLBACK_CHAIN=

# data
RDBMS_URL=
VECTOR_DB_URL=
VECTOR_DB_BACKEND=
EMBEDDING_MODEL=

# automation
BROWSER_AUTOMATION_BACKEND=camoufox
BROWSER_FALLBACK_BACKEND=playwright
IMAGE_RUNTIME_BASE_URL=
IMAGE_RUNTIME_MODE=on-demand

# agent instances
HERMES_MAIN_PORT=
HERMES_SCHEDULER_PORT=
SCHEDULER_MODE=hermes-cron

# notifications (Hermes 메시징 게이트웨이 설정)
TELEGRAM_BOT_TOKEN=
TELEGRAM_CHAT_ID=
DISCORD_WEBHOOK_URL=
SLACK_WEBHOOK_URL=

# backup
BACKUP_TARGET_NODE_B_PATH=
BACKUP_OFFSITE_PATH=
BACKUP_ENCRYPTION_KEY_REF=

# watchdog
WATCHDOG_CHECK_INTERVAL_SEC=60
WATCHDOG_MAX_RESTART_COUNT=5
WATCHDOG_ALERT_CHANNEL=
```

규칙:

- 코드에는 제품명보다 **슬롯명**을 쓴다.
- 제품 교체 시 `.env` 와 배포 스크립트만 바꾸면 되게 설계한다.
- 모델명 변경은 애플리케이션 코드 수정 사유가 되면 안 된다.
- Hermes 자체 설정(config.yaml 등)과 `.env`를 병행하되, 슬롯명은 `.env`에서 관리한다.
- LAN IP는 메쉬 네트워크 폴백용으로 반드시 등록한다.

## 9. 구축 순서

### 9.1 1단계: Node B 먼저 준비

이유: 제어 노드가 추론 노드에 의존하므로, 먼저 계산 노드를 안정화하는 편이 전체 검증이 쉽다.

권장 순서:

1. Tailscale 연결 + LAN IP 확인 및 등록
2. 로컬 모델 런타임 설치
3. `MODEL_FAST`, `MODEL_ORCHESTRATOR` 슬롯에 해당하는 모델 준비
4. 내부 UI 연결
5. Hermes 내장 Camoufox 확인, 필요 시 Playwright MCP 설치
6. 이미지 런타임(선택) 설치
7. Node A 에서 API 호출 테스트 (Tailscale + LAN 양쪽 경로)

### 9.2 2단계: Node A 준비

권장 순서:

1. Tailscale 연결 + LAN IP 확인 및 등록
2. PostgreSQL 설치
3. 필요 시 MySQL 추가
4. 벡터 저장소 설치
5. Hermes Agent 설치 및 설정
6. hermes-main 인스턴스 launchd plist 구성 및 시작
7. hermes-scheduler 인스턴스 launchd plist 구성 및 시작
8. 워치독 스크립트 구성 및 launchd 등록
9. Claude Code / MCP 연결 (Hermes MCP 서버 노출 확인)
10. 외부 API 프로바이더 설정 및 폴백 체인 구성
11. 메시징 게이트웨이 연결 (Telegram, Discord, Slack 등)
12. Node B로의 백업 동기화 스크립트 구성

### 9.3 3단계: 워크플로우 검증

최소 검증 순서:

1. Node A 가 Node B 의 모델 런타임을 호출할 수 있는지 확인 (Tailscale 경로)
2. Tailscale 비활성화 후 LAN 폴백 경로로 동일 호출 성공하는지 확인
3. Node A 가 DB와 벡터 저장소에 읽기/쓰기를 할 수 있는지 확인
4. Hermes 브라우저 자동화(Camoufox) 작업이 성공적으로 동작하는지 확인
5. 코드 수정 → 테스트 → 재배포 흐름이 되는지 확인 (Claude Code ↔ Hermes MCP)
6. 메시징 게이트웨이 알림이 정상 송신되는지 확인
7. 스킬 생성 및 에피소딕 메모리 기록이 정상인지 확인
8. 워치독이 에이전트 프로세스 비정상 종료를 감지하고 재시작하는지 확인
9. Node B 다운 시 외부 API 폴백이 동작하는지 확인
10. 백업 동기화가 Node B로 정상 전송되는지 확인

## 10. 표준 워크플로우

### 10.1 개발 및 배포 자동화

```text
사용자 요청
  → hermes-main 수신
  → 작업 유형 판별 (MODEL_FAST)
  → Claude Code / MCP 호출
  → 코드 수정 및 테스트
  → 배포 스크립트 실행
  → 결과 저장 (PostgreSQL)
  → 스킬 자동 생성 (반복 패턴 감지 시)
  → 메시징 게이트웨이로 알림 발송
```

### 10.2 주기적 데이터 수집 및 분석

```text
hermes-scheduler Cron 트리거
  → hermes-main 호출
  → Camoufox로 대상 사이트 접근
  → 수집 결과 정리
  → MODEL_FAST 또는 MODEL_ORCHESTRATOR 로 1차 분석
  → 필요 시 MODEL_DEEP 또는 MODEL_EXTERNAL_TEXT 로 승격
  → DB / 벡터 저장소 반영
  → 에피소딕 메모리에 작업 맥락 저장
  → 메시징 게이트웨이로 알림 발송
```

### 10.3 장문 문서 처리

```text
문서 입력
  → hermes-main 전처리
  → 로컬 중형 모델로 1차 구조화
  → 문맥이 길거나 비용 효율이 더 좋으면 외부 장문 모델 호출
  → 요약 / 태깅 / 추출 결과 저장
  → 후속 액션 생성
```

### 10.4 이미지 생성

```text
이미지 요청
  → hermes-main 수신
  → Node B에 Image Runtime 기동 (SSH 터미널 백엔드)
  → 워크플로우 실행
  → 결과 저장 또는 전송
  → Image Runtime 종료
```

## 11. 리소스 정책

### 11.1 Node A (Mac Mini M4 16GB) 정책

- 로컬 대형 LLM을 직접 구동하지 않는다.
- 기본 RDBMS 는 PostgreSQL 하나로 시작한다.
- MySQL은 특정 서비스 호환성 요구가 있을 때만 추가한다.
- swap 이 지속적으로 증가하면 벡터 저장소 또는 부가 워커를 Node B 로 이동한다.
- Hermes 인스턴스는 최대 2개(main + scheduler)를 상시 유지하고, 3번째 이상은 필요 시만 기동한다.
- 제어 노드에서 가장 중요한 것은 **서비스 생존성**이다.

### 11.2 Node B (Mac Studio M4 Max 128GB) 정책

- 중형 오케스트레이터 모델은 상시 대기 가능하다.
- 70B급 대형 분석 모델은 필요 시만 로드한다.
- 이미지 생성 런타임은 요청 시만 기동한다.
- 브라우저 자동화와 대형 모델 추론이 동시에 많아질 때는 아래 우선순위에 따라 직렬화한다:
  1. 진행 중인 브라우저 자동화 작업 완료 (중단 비용이 높음)
  2. 오케스트레이터 모델 추론
  3. 대형 분석 모델 추론
  4. 이미지 생성 (가장 낮은 우선순위)

### 11.3 승격/이관 트리거

아래 중 하나라도 성립하면 구성 요소 이관을 검토한다.

- Node A 에서 swap 이 장시간 지속 (30분 이상 연속)
- 벡터 저장소 쿼리 응답이 500ms를 지속적으로 초과
- 벡터 컬렉션이 100만 건을 초과하거나 디스크 사용이 Node A 전체의 50% 이상
- 인증/권한 관리가 단순 프록시로 감당되지 않음
- 브라우저 자동화가 장시간 상주하며 메모리 압박 유발
- 대형 모델 호출 빈도가 높아 상시 로드가 더 유리해짐

## 12. 스킬 관리 정책

### 12.1 스킬 저장 기준

다음 조건 중 하나 이상을 만족하는 작업 패턴을 스킬로 저장한다.

- 동일 유형의 작업이 3회 이상 반복됨
- 작업 단계가 3단계 이상이며 매번 동일한 순서로 실행됨
- 특정 도구 조합이 반복적으로 사용됨
- 사용자가 명시적으로 스킬 저장을 요청함

### 12.2 스킬 검증

- 새 스킬은 생성 후 최초 3회 실행에서 성공 여부를 추적한다.
- 3회 중 2회 이상 실패하면 해당 스킬을 비활성화하고 로그에 기록한다.
- 스킬 내부에서 모델 전환(`/model`)을 사용하는 경우, 전환 대상 모델의 가용성을 사전 확인하는 단계를 포함한다.

### 12.3 스킬 정리

- 90일간 미사용된 스킬은 비활성 목록으로 이동한다.
- 비활성 목록의 스킬은 추가 90일 후 아카이브(삭제가 아닌 별도 보관)한다.
- 스킬 정리는 hermes-scheduler의 월간 Cron 작업으로 자동화한다.

### 12.4 스킬 백업

- 스킬 DB는 PostgreSQL 덤프와 동등한 우선순위로 백업한다.
- 백업 주기: 일 1회 이상.

### 13.1 보존 기간

- 브라우저 자동화에서 수집된 원시 HTML 전체를 에피소딕 메모리에 저장하지 않는다. 추출된 구조화 데이터만 저장한다.

## 14. 장애 폴백 매트릭스

### 14.1 Node A ↔ Node B 네트워크 단절

| 단계 | 동작 |
|---|---|
| 감지 | hermes-main이 Node B 모델 런타임 healthcheck 실패 (3회 연속, 10초 간격) |
| 1차 폴백 | Tailscale → LAN IP 경로로 전환하여 재시도 |
| 2차 폴백 | LAN도 실패 시, `MODEL_EXTERNAL_TEXT` (외부 API)로 추론 작업 우회 |
| 제한 | 이미지 생성, 로컬 대형 모델 분석은 불가. 해당 작업은 큐에 보관 |
| 복구 | 네트워크 복구 감지 시 자동으로 로컬 경로 복원, 큐의 보류 작업 순차 실행 |
| 알림 | 네트워크 단절 감지 즉시 메시징 게이트웨이로 운영자에게 알림 |

### 14.2 Node B 완전 다운

| 단계 | 동작 |
|---|---|
| 감지 | 네트워크 단절과 동일한 healthcheck + SSH 연결 시도 실패 |
| 폴백 | 모든 추론 작업을 `MODEL_EXTERNAL_TEXT`로 전환 |
| 제한 | 이미지 생성 불가, 브라우저 자동화 불가, 로컬 모델 분석 불가 |
| 보류 | Node B 전용 작업(이미지, 브라우저)은 실패 기록 후 큐에 보관 |
| 알림 | 즉시 운영자에게 알림. 수동 개입 필요 표시 |

### 14.3 Hermes 프로세스 비정상 종료

| 단계 | 동작 |
|---|---|
| 감지 | launchd가 프로세스 종료 감지 |
| 1차 복구 | launchd `KeepAlive`에 의해 자동 재시작 |
| 반복 크래시 | 워치독이 5분 내 3회 이상 재시작 감지 시 크래시 루프로 판정 |
| 크래시 루프 대응 | 재시작 중지, 에이전트 외부 채널(직접 메일/SMS)로 운영자에게 알림 |
| 수동 복구 | 운영자가 로그 확인 후 원인 해결, 수동 재시작 |

### 14.4 모델 로드 실패

| 단계 | 동작 |
|---|---|
| 감지 | 모델 런타임 API가 에러 반환 또는 타임아웃 |
| 1차 폴백 | Hermes 프로바이더 라우터의 폴백 체인 자동 동작 (다음 프로바이더로 전환) |
| 2차 폴백 | 모든 로컬 프로바이더 실패 시 `MODEL_EXTERNAL_TEXT`로 전환 |
| 기록 | 실패한 모델, 에러 메시지, 타임스탬프를 구조화 로그에 기록 |

### 14.5 외부 API 제한/장애

| 단계 | 동작 |
|---|---|
| 감지 | HTTP 429 (Rate Limit) 또는 5xx 에러 |
| 대응 | 지수 백오프 재시도 (최대 3회, 초기 대기 5초) |
| 폴백 | 외부 API 전체 불통 시 로컬 `MODEL_DEEP` 로드하여 처리 |
| 기록 | 외부 API 사용량, 에러율을 일별 로그로 집계 |

## 15. 데이터 저장 정책

### 15.1 기본 원칙

- 원본 데이터는 가능하면 관계형 DB 또는 파일 스토리지에 저장
- 벡터 저장소는 **검색용/보조 기억용**으로 사용 (원본이 아닌 파생 데이터)
- 에이전트의 장기 기억(스킬 DB, 에피소딕 메모리)은 제어 노드에 집중
- 추론 노드는 캐시와 임시 산출물만 보관

### 15.2 데이터 분류별 저장 규칙

| 데이터 유형 | 저장 위치 | 보존 정책 |
|---|---|---|
| 작업 이력, 결과 메타데이터 | PostgreSQL | 영구 (용량 관리 시 연 단위 아카이브) |
| 수집된 원본 문서/파일 | 파일 스토리지 (Node A) | 영구 |
| 벡터 임베딩 | 벡터 저장소 | 원본 존재하는 한 재생성 가능. 임베딩 모델 변경 시 전체 재색인 |
| 에이전트 스킬 | Hermes SQLite | 영구 (90일 미사용 시 비활성화, 추가 90일 후 아카이브) |
| 에피소딕 메모리 | Hermes SQLite | 보존 기간 후 아카이브 |
| 브라우저 자동화 세션 | Node B 임시 | 작업 완료 후 삭제 |
| 이미지 생성 중간 산출물 | Node B 임시 | 최종 결과만 Node A로 전송 후 삭제 |
| 로그 | 파일 스토리지 (Node A) | 90일 보존, 이후 압축 아카이브 |

### 15.3 민감 데이터 처리

- 외부 사이트에서 수집한 원시 HTML 전체를 무조건 저장하지 않는다.
- 개인정보, 인증정보가 포함된 수집물은 필요한 필드만 추출하고 나머지는 폐기한다.
- 로그에 API 키, 토큰, 비밀번호가 남지 않도록 로깅 시점에 마스킹한다.
- 마스킹 대상 패턴: API 키, Bearer 토큰, 비밀번호 필드, 신용카드 번호, 주민등록번호 형식

### 15.4 벡터 저장소 운영

초기 정책:

- 경량 스토어로 시작
- 단일 사용자 / 내부망 / 소규모 컬렉션 기준 최적화

승격 트리거:

- 컬렉션 문서 수 100만 건 초과
- 쿼리 응답 500ms 지속적 초과
- 인증/권한 분리 요구 발생
- 원격 접근 요건 발생

승격 시 더 강한 운영형 스토어(Qdrant 등)로 전환한다.

## 16. 보안 체크리스트

### 16.1 반드시 지킬 것

- 내부 서비스 포트는 기본적으로 외부 공개 금지
- 모델 런타임과 벡터 저장소는 Tailscale 내부에서만 접근
- UI는 최초 관리자 계정 생성 후 공개 가입 비활성화
- DB 바인딩 범위 제한
- SSH 키 기반 접근 사용
- 웹훅/토큰/비밀값은 `.env` 와 비밀 저장소에 보관

### 16.2 정기 보안 점검

| 점검 항목 | 주기 | 설명 |
|---|---|---|
| SSH 키 교체 | 6개월 | 모든 노드의 SSH 키 갱신 |
| API 토큰/웹훅 URL 로테이션 | 3개월 | 외부 API 키, 봇 토큰, 웹훅 URL 갱신 |
| Tailscale ACL 검토 | 3개월 | 불필요한 접근 경로 제거 |
| DB 접근 계정 검토 | 6개월 | 사용하지 않는 계정 비활성화 |
| 백업 복구 테스트 | 3개월 | 백업에서 실제로 복구되는지 검증 |
| 로그 민감정보 마스킹 확인 | 1개월 | 최근 로그에서 미마스킹 데이터 점검 |
| launchd/서비스 권한 검토 | 6개월 | 불필요한 권한 상승 여부 확인 |

### 16.3 권장 사항

- 관리자 단말도 Tailscale 을 통해서만 내부 접속
- 리버스 프록시를 둘 경우 TLS 와 기본 인증 또는 SSO 적용
- 관리자 접근에 MFA 적용 (가능한 경우)
- 백업 파일은 암호화하여 저장
- 브라우저 자동화용 계정은 전용 계정 사용, 세션 토큰은 암호화 저장

| 디스크 사용량 | 필수 | 필수 | 시스템 모니터링 스크립트 |
| 에이전트 프로세스 상태 | 필수 | 선택 | 워치독 + launchd |
| 모델 런타임 응답 시간 | 선택 | 필수 | healthcheck 스크립트 |
| 브라우저 자동화 실패율 | 선택 | 필수 | 작업 로그 집계 |
| DB 연결 수 / 지연 | 필수 | 해당 없음 | PostgreSQL 통계 뷰 |
| 알림 전송 성공률 | 필수 | 선택 | 메시징 게이트웨이 로그 |
| 스킬 실행 성공/실패율 | 필수 | 선택 | Hermes 작업 로그 |
| 에피소딕 메모리 DB 크기 | 필수 | 해당 없음 | 주간 Cron 스크립트 |

### 17.2 구조화 로그 요구사항

모든 작업 로그는 다음 필드를 포함해야 한다.

- `job_id`: 고유 작업 식별자
- `timestamp`: ISO 8601 형식
- `job_type`: 작업 유형 (분류, 분석, 브라우저, 이미지 등)
- `model_slot`: 사용된 모델 슬롯 (MODEL_FAST, MODEL_DEEP 등)
- `status`: 성공/실패/타임아웃/폴백
- `duration_ms`: 소요 시간
- `error_type`: 실패 시 에러 유형
- `fallback_used`: 폴백 경로 사용 여부 및 대상

### 17.3 모니터링 도구 기본값

초기에는 다음 조합으로 시작한다.

- 시스템 메트릭: 쉘 스크립트 + Cron으로 메모리/디스크/프로세스 상태 수집 → 로그 파일 기록
- 알림 임계치 도달 시 메시징 게이트웨이로 즉시 알림
- 확장 필요 시 Prometheus + Grafana 스택 도입 검토

### 17.4 점검 규칙

- Node A 에서 swap 이 의미 있게 쌓이면(30분 이상 연속) 먼저 부가 서비스를 줄인다.
- Node B 에서 응답 지연이 커지면 상시 로드 모델 수를 줄인다.
- 브라우저 자동화 실패가 반복되면 모델 문제보다 사이트 변화/차단을 먼저 의심한다.
- 이미지 런타임이 종료되지 않으면 가장 먼저 orphan process 여부를 본다.
- 스킬 실패율이 급증하면 최근 모델 변경 또는 외부 서비스 변경을 먼저 확인한다.

## 18. 백업 및 복구

### 18.1 백업 대상 및 주기

| 백업 대상 | 주기 | 저장 위치 | 비고 |
|---|---|---|---|
| PostgreSQL dump | 일 1회 | Node B + 오프사이트 | 암호화 필수 |
| MySQL dump (사용 시) | 일 1회 | Node B + 오프사이트 | 암호화 필수 |
| Hermes 스킬 DB (SQLite) | 일 1회 | Node B + 오프사이트 | 핵심 학습 자산 |
| Hermes 에피소딕 메모리 DB (SQLite) | 일 1회 | Node B + 오프사이트 | 핵심 학습 자산 |
| 벡터 저장소 데이터 | 주 1회 | Node B | 원본에서 재생성 가능하므로 우선순위 낮음 |
| `.env`, 배포 스크립트, launchd plist | 변경 시 | Git 저장소 + 오프사이트 | 버전 관리 |
| Hermes 설정 파일 (config.yaml 등) | 변경 시 | Git 저장소 + 오프사이트 | 버전 관리 |
| 에이전트 작업 로그 | 주 1회 (90일 이상분 압축) | 오프사이트 | 용량 관리 |
| 핵심 워크플로우 정의 파일 | 변경 시 | Git 저장소 | 버전 관리 |

### 18.2 백업 암호화

- 모든 오프사이트 백업은 암호화하여 전송 및 저장한다.
- 암호화 키는 `.env`의 `BACKUP_ENCRYPTION_KEY_REF`로 참조하며, 키 자체는 별도 비밀 저장소에 보관한다.

### 18.3 복구 우선순위

1. Node A 의 DB(PostgreSQL)와 설정 복구
2. Hermes 스킬 DB + 에피소딕 메모리 DB 복구
3. Hermes Agent 인스턴스 복구 (hermes-main → hermes-scheduler 순)
4. Node B 모델 런타임 복구
5. 브라우저 자동화 및 이미지 런타임 복구
6. 벡터 저장소 복구 (필요 시 원본에서 재색인)
7. UI 복구

즉, **기억과 학습 자산을 먼저 살리고, 제어를 복원한 다음, 계산은 나중에 붙인다**.

### 18.4 복구 테스트

- 3개월마다 백업에서 실제 복구를 수행하여 검증한다.
- 검증 항목: DB 무결성, 스킬 DB 로드 가능 여부, 에이전트 정상 시작 여부, 설정 파일 일관성

## 19. 변경 관리

### 19.1 변경 시 원칙

- 모델 교체는 슬롯명 유지 하에 수행
- 브라우저 자동화 백엔드 교체 시 에이전트 인터페이스는 유지
- DB 제품 교체 시 애플리케이션 접속 계층을 먼저 분리
- 새 서비스를 추가할 때는 항상 "상시 실행이 필요한가?" 를 먼저 묻는다
- 에이전트 런타임 교체 시 스킬 DB와 에피소딕 메모리의 마이그레이션 계획을 먼저 수립한다

### 19.2 롤백 기준

아래 중 하나라도 발생하면 직전 안정 상태로 롤백한다.

- 제어 노드 메모리 압박 증가
- 배포 후 에이전트 작업 실패율 급증
- 모델 호출 지연이 운영 임계치를 넘음
- 브라우저 자동화 성공률 급락
- 알림 또는 DB 쓰기 실패
- 스킬 실행 성공률 급락

## 21. 에이전트 실행 지침

에이전트는 아래 원칙을 따른다.

1. 문서에 나온 모델명을 **예시**로 해석할 것
2. 항상 **역할 슬롯**을 통해 모델을 선택할 것
3. 제어 노드의 가용성을 추론 품질보다 우선할 것
4. 이미지 생성과 대형 분석은 **상시 실행 금지**
5. 내부 서비스는 **사설 메쉬 네트워크 기준**으로만 연결할 것 (LAN 폴백 허용)
6. 새로운 서비스 추가 전, 기존 워크플로우로 흡수 가능한지 먼저 판단할 것
7. 반복되는 작업 패턴은 **스킬로 저장**하여 학습 자산을 축적할 것
8. 에피소딕 메모리에 **민감 정보를 마스킹** 없이 저장하지 않을 것
9. 모든 작업에 **job_id를 부여**하고 구조화 로그를 남길 것
10. 실패 시 **폴백 체인**을 따르고, 최종 실패는 운영자에게 알릴 것
11. 모델 전환 패턴이 안정되면 해당 패턴을 **스킬로 내재화**할 것

## 22. 한 줄 요약

운영 관점에서 이 인프라는 **Node A 가 기억하고 제어하고, Node B 가 계산하고 생성하는 구조**다.  
에이전트는 작업을 수행하며 **스킬과 기억을 축적**하고, 모든 구현은 특정 모델명이 아니라 **역할 슬롯, 자원 예산, 보안 경계, 장애 폴백**을 기준으로 유지한다.
