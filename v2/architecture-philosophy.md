# 시스템 아키텍처 철학서
## 하이브리드 로컬 AI 인프라 (Mac Mini + Mac Studio 2-노드 기준)

> 이 문서는 **설치 순서**가 아니라 **설계 철학과 의사결정 기준**을 정의한다.  
> 에이전트(Hermes, Claude 계열, 기타 자동화 에이전트)는 이 문서를 읽고 **왜 이 구조를 쓰는지**, **무엇을 고정하고 무엇을 교체 가능한지**를 먼저 이해해야 한다.

---

## 1. 문서 목적

이 인프라는 다음 조건을 동시에 만족시키기 위한 구조다.

- 작은 상시 가동 노드와 큰 추론 노드를 조합해 **비용·전력·가용성**을 균형 있게 맞춘다.
- 개발 자동화, 데이터 수집, 분석, 브라우저 작업, 이미지 생성을 **하나의 운영 체계**로 묶는다.
- 인터넷에 직접 노출하지 않고 **프라이빗 네트워크 우선**으로 운영한다.
- 특정 모델이나 벤더에 묶이지 않고, **역할 중심(role-based)** 으로 부품을 교체할 수 있게 한다.
- 에이전트 런타임(현재 Hermes Agent)이 제공하는 생태계를 우선 활용하되, 보조 스크립트와 도구는 언어에 구애받지 않는다.
- 에이전트가 작업을 수행할수록 **스킬과 기억이 누적**되어 시스템이 복리적으로 개선되는 구조를 지향한다.

## 2. 이 문서가 고정하는 것과 고정하지 않는 것

### 2.1 고정하는 것

- **노드 역할 분리**
  - 소형 노드 = 제어·상태·지속성
  - 대형 노드 = 추론·자동화·생성
- **네트워크 경계**
  - 내부 통신은 사설 메쉬 네트워크 기준
  - 서비스 포트는 외부 공개를 기본 금지
  - 메쉬 네트워크 장애 시 로컬 LAN 폴백 경로 확보
- **운영 원칙**
  - 항상 켜둘 것과 필요 시만 켤 것을 구분
  - 에이전트는 도구와 모델을 역할에 따라 라우팅
  - 에이전트의 학습 자산(스킬, 에피소딕 메모리)은 영속 데이터로 보호
- **의사결정 우선순위**
  1. 보안 경계
  2. 가용성 (제어 노드 생존성)
  3. 에이전트 학습 자산 보존
  4. 메모리/자원 예산
  5. 자동화 단순성
  6. 개별 모델/도구 선택

### 2.2 고정하지 않는 것

- 특정 모델명
- 특정 벡터 DB 제품명
- 특정 UI 제품명
- 특정 브라우저 자동화 프레임워크 버전
- 특정 외부 API 모델 버전명
- 특정 에이전트 런타임 (현재 Hermes Agent가 최선이지만 동일 역할을 수행하는 대체재로 교체 가능)

문서 안에 나오는 제품/모델명은 **현재 환경에서 적절한 예시**일 뿐, **구조적 고정값**이 아니다.  
동일 역할을 더 잘 수행하는 대체재가 나오면 교체 가능하다.

## 3. 핵심 설계 철학

### 3.1 Control Plane 과 Compute Plane 의 분리

시스템 제어, 스케줄링, DB, 알림, 코드 배포 같은 **운영 핵심 기능**은 가벼운 노드에 둔다.  
대규모 LLM 추론, 브라우저 자동화, 이미지 생성처럼 **자원 소비가 큰 기능**은 별도 노드로 분리한다.

이 원칙의 목적은 다음과 같다.

- 큰 모델이 메모리를 많이 먹어도, 핵심 운영 서비스는 계속 살아 있게 한다.
- 추론 노드가 재시작되거나 모델이 바뀌어도, 제어 노드는 영향을 덜 받는다.
- 장애 분석 시 문제의 성격을 빨리 분리할 수 있다.

### 3.2 Stateful 과 Stateless 의 분리

DB, 벡터 스토어, 작업 로그, 에이전트 메모리(에피소딕 메모리, 스킬 DB)처럼 **상태를 보존해야 하는 것**은 제어 노드에 둔다.  
브라우저 세션, 일시적 분석 작업, 대형 모델 로드, 이미지 생성 작업처럼 **작업 후 버려질 수 있는 것**은 추론 노드에 둔다.

### 3.3 Local-first, Cloud-optional

가능하면 로컬에서 끝내고, 아래 조건에만 외부 API를 쓴다.

- 긴 컨텍스트/대규모 문서 처리에서 외부 API가 더 경제적일 때
- 특정 기능(예: 최신 멀티모달, 매우 긴 문맥, 특정 SaaS 연동)이 로컬보다 확실히 유리할 때
- 장애 시 우회 경로가 필요할 때 (Compute Node 다운 시 외부 API 자동 폴백 포함)

즉, 외부 API는 기본 경로가 아니라 **확장 슬롯이자 폴백 경로**다.

### 3.4 Always-on 최소화, On-demand 최대화

항상 켜두는 서비스는 아래 성격을 가져야 한다.

- 작고 안정적이어야 한다.
- 시스템 전체의 제어권을 가져야 한다.
- 재시작 시 복구가 단순해야 한다.

반대로 무거운 모델, 이미지 생성 런타임, 특수 분석 런타임은 **요청 시 기동 후 종료**를 기본으로 한다.

### 3.5 Replaceable Components

모든 핵심 구성 요소는 아래 형태로 다뤄야 한다.

- **역할**로 식별: 예) "메인 오케스트레이터용 중형 로컬 LLM"
- **접점**으로 연결: 예) HTTP API, MCP, SSH, DB URL
- **정책**으로 선택: 예) 지연 시간, 메모리, 정확도, 비용

이 원칙을 지키면 모델이나 제품을 바꿔도 상위 구조는 유지된다.

### 3.6 Agent-centric 운영과 자기 개선

별도 워크플로우 툴을 계속 늘리기보다, **에이전트가 오케스트레이터 역할을 직접 수행**하는 것을 기본값으로 둔다.

- 태스크 수신
- 모델 라우팅
- 도구 호출
- 결과 저장
- 알림 전송
- 실패 재시도

여기에 더해, 에이전트는 **자기 개선 루프**를 갖는 것을 핵심 설계 원칙으로 한다.

- 완료된 작업에서 재사용 가능한 **스킬을 자동 생성·저장**한다.
- **에피소딕 메모리**로 과거 작업 맥락을 누적 참조한다.
- 시간이 지날수록 사용자 환경에 최적화되는 **복리형 자산**을 축적한다.

이 자기 개선 루프는 단순 자동화와 이 인프라를 구분짓는 핵심 가치다.  
스킬과 메모리는 모델과 달리 교체 시 이전이 어려우므로 **영속 데이터와 동등하게 보호**한다.

기본값은 **간단한 구조**이며, 운영 복잡도가 일정 임계치를 넘을 때만 전용 오케스트레이션 계층을 추가한다.

### 3.7 실패는 정상 경로다

에이전트는 실패를 예외가 아니라 정상 상태로 취급해야 한다.

- 모델 타임아웃
- 브라우저 차단
- 외부 API 제한
- DB 연결 실패
- 메모리 부족
- 노드 간 네트워크 단절

따라서 모든 워크플로우는 다음을 기본으로 고려해야 한다.

- **재시도**: 일시적 실패에 대해 지수 백오프 기반 재시도
- **대체 모델/경로**: 로컬 모델 실패 시 외부 API 폴백
- **부분 결과 저장**: 중간 산출물을 잃지 않도록 단계별 영속화
- **멱등성(Idempotency)**: 동일 작업을 재실행해도 부작용이 없는 설계
- **실패 기록**: 모든 실패를 구조화된 로그로 남겨 사후 분석 가능

## 4. 현재 하드웨어에 대한 역할 매핑

이 철학은 다양한 하드웨어 조합에 적용될 수 있지만, 현재 기준 매핑은 아래와 같다.

| 노드 | 현재 하드웨어 | 철학적 역할 |
|---|---|---|
| Node A | Mac Mini M4 16GB | Control / Data Node |
| Node B | Mac Studio M4 Max 128GB | Inference / Automation Node |

### 4.1 Node A: Control / Data Node

이 노드는 다음을 책임진다.

- 에이전트 실행과 스케줄링
- 코드 수정·배포 제어
- DB 및 벡터 스토어
- 에이전트 학습 자산(스킬 DB, 에피소딕 메모리)의 영속화
- 작업 결과의 영속화
- 메시징 게이트웨이를 통한 알림
- 외부 API 호출의 중앙화
- 시스템 상태 추적

이 노드는 **늘 살아 있어야 하는 노드**다.  
이 노드가 시스템의 **단일 장애점(SPOF)** 이라는 점은 의도적 트레이드오프다. HA 클러스터는 비목표이므로, 대신 다음으로 위험을 완화한다.

- Node A의 핵심 데이터(DB 덤프, 스킬 DB, 에피소딕 메모리, 설정)를 Node B에 주기적으로 동기화
- 외부 저장소(NAS, S3 등)로의 오프사이트 백업
- 에이전트 프로세스에 대한 워치독 감시

### 4.2 Node B: Inference / Automation Node

이 노드는 다음을 책임진다.

- 로컬 LLM 추론
- 브라우저 자동화
- 대형 문맥 분석
- 이미지 생성
- 실험적·고비용 계산 작업

이 노드는 **계산을 담당하는 노드**다.  
필요 시 재시작되거나 모델이 바뀔 수 있으며, 그 자체가 시스템의 단일 진실 저장소가 되어서는 안 된다.

## 5. 모델 및 도구 선택 철학

### 5.1 모델/도구명보다 "역할"을 먼저 본다

에이전트는 아래 질문 순서로 판단해야 한다.

1. 이 작업은 **분류/파싱**인가?
2. 이 작업은 **오케스트레이션 / tool calling**인가?
3. 이 작업은 **깊은 분석 / 긴 문맥**인가?
4. 이 작업은 **코드 작업**인가?
5. 이 작업은 **멀티모달 생성**인가?
6. 이 작업은 **외부 API가 더 적합한가**?

그 뒤에야 구체적인 모델이나 런타임을 고른다.

### 5.2 역할별 모델 계층

아래 계층은 고정 규격이 아니라 선택 기준이다.

| 역할 | 요구 특성 | 현재 환경에서 적절한 예시 |
|---|---|---|
| 경량 모델 | 빠른 응답, 저메모리, 분류/파싱/라우팅 | 4B~8B급 instruct 모델 |
| 중형 오케스트레이터 | tool calling, 일반 추론, 멀티스텝 판단 | 20B~35B급 instruct 또는 MoE 계열 |
| 대형 분석 모델 | 긴 컨텍스트, 복잡한 분석, 높은 일관성 | 70B급 instruct 모델 또는 동급 |
| 코드 특화 도우미 | 코드 수정, 리팩토링, 저장소 조작 | Claude Code 또는 코드 특화 로컬 모델 |
| 외부 장문 처리 모델 | 장문 요약, 대규모 문서 처리, 비용 대비 성능 | 장문 컨텍스트가 강한 stable cloud 모델 |
| 이미지 생성 모델 | 이미지 합성/편집, 워크플로우 구성 | Flux / SDXL 계열과 호환되는 런타임 |

### 5.3 현재 환경에 대한 추천 해석

현재 하드웨어 구성에서는 다음 해석이 실용적이다.

- **경량 역할**: 작은 분류/요약/라우팅 전용 모델이 적합하다.
- **중형 오케스트레이터 역할**: 메모리 대비 성능이 좋은 20B~30B급 모델이 적합하다.
- **깊은 분석 역할**: 70B급 모델은 충분한 메모리를 가진 노드에서만 필요 시 로드한다.
- **코드 역할**: 로컬 모델만으로 고집하기보다, 코드 에이전트와 MCP를 우선 고려한다.
- **장문/대규모 처리 역할**: 비용과 속도가 괜찮은 외부 API를 보조 슬롯으로 둔다.
- **이미지 역할**: 상시 실행이 아니라 on-demand 런타임이 적합하다.

핵심은 **"항상 큰 모델"** 이 아니라 **"작업에 맞는 가장 작은 충분한 모델"** 이다.

### 5.4 모델 라우팅 전략

역할 슬롯 체계를 실제 에이전트가 사용하기 위해 두 단계 접근을 취한다.

**1단계 — 명시적 라우터 레이어 (초기 구현)**

작업 수신 시 경량 모델(MODEL_FAST)로 작업 유형을 분류하고, 분류 결과에 따라 적절한 모델 슬롯을 지정하여 에이전트에 전달한다. 수십 줄 규모의 스크립트로 구현 가능하다.

**2단계 — 스킬 기반 자체 학습 (성숙 단계)**

에이전트의 런타임 모델 전환 기능을 스킬 안에서 활용한다. "이 유형의 작업이면 먼저 모델을 X로 전환하고 실행하라"는 패턴을 스킬로 학습시키면, 에이전트 자체가 역할 라우팅을 내재화한다. 이 접근이 자기 개선 루프의 장점을 가장 잘 살리는 방향이다.

두 접근은 순차적으로 적용한다.

## 6. 에이전트 런타임 철학

### 6.1 현재 선택: Hermes Agent

현재 메인 에이전트 런타임으로 Hermes Agent(NousResearch/hermes-agent)를 채택한다.

채택의 핵심 근거는 다음과 같다.

- **자기 개선 루프**: 스킬 자동 생성 + 에피소딕 메모리(SQLite FTS5). 커스텀 빌드나 다른 프레임워크로는 이 루프를 처음부터 만들어야 하며, 그 자체가 별도 프로젝트 규모다.
- **MCP 양방향 통합**: MCP 클라이언트(stdio + HTTP)이자 MCP 서버로서 Claude Code 등과 직접 연동된다.
- **40+ 내장 도구**: 웹 검색, 브라우저 자동화(Camoufox stealth), 코드 샌드박스, 파일 작업, 비전, TTS, 이미지 생성, 하위 에이전트 위임 등.
- **프로바이더 라우터**: 8+ 프로바이더 지원, 순서 지정 폴백 체인, 런타임 모델 전환.
- **14개 플랫폼 메시징 게이트웨이**: Telegram, Discord, Slack, WhatsApp, Signal, Matrix, 이메일, SMS 등.
- **터미널 백엔드**: 로컬, Docker, SSH 등. 2-노드 구조에서 SSH 백엔드로 Node B 원격 제어 가능.

### 6.2 교체 가능성

Hermes Agent는 현재 이 역할에 최선이지만, 역할 기반 교체 가능성은 유지한다. 교체 시 다음을 충족해야 한다.

- 자기 개선 루프(스킬 + 에피소딕 메모리) 동등 기능
- MCP 양방향 통합
- 멀티 프로바이더 폴백
- 메시징 게이트웨이

### 6.3 알려진 제약과 수용

| 제약 | 수용 근거 |
|---|---|
| Python 기반 | Hermes가 40+ 도구/14개 메시징/MCP/자기개선을 모두 해결하므로, 이를 다른 언어로 재구현할 이유 없음 |
| 동기식 코어 루프 (인스턴스당 1대화) | 역할별 인스턴스 분리로 해결 |
| 멀티노드 오케스트레이션 없음 | SSH 터미널 백엔드 + MCP로 Node B 제어 |

### 6.4 언어 정책

기존 "Node.js/JavaScript 중심" 정책을 다음으로 변경한다.

- **에이전트 런타임 생태계를 우선** 활용한다. 현재 Hermes Agent는 Python 기반이다.
- **보조 스크립트와 도구는 언어에 구애받지 않는다.** 가장 적합한 도구를 쓴다.
- **Python이 기본이 되는 영역**: 에이전트 런타임, ML/AI 도구(ComfyUI 등), 브라우저 자동화(Camoufox), 데이터 처리 파이프라인.
- **언어 무관 영역**: 유틸리티 스크립트, 웹훅 핸들러, 간단한 API 어댑터, 설정 자동화.

## 7. 데이터와 메모리 철학

### 7.1 데이터 중력(Data Gravity)

작업 결과, 요약, 임베딩, 운영 로그, 에이전트 스킬, 에피소딕 메모리는 제어 노드 쪽에 모아야 한다.

- 장애 복구가 단순해진다.
- 여러 에이전트 인스턴스가 같은 기억 저장소를 참조할 수 있다.
- 추론 노드 교체 시 이전 맥락을 유지하기 쉽다.

### 7.2 메모리 예산 기반 설계

이 인프라는 **CPU 성능보다 메모리 제약**이 더 큰 설계 변수다.

- 소형 노드는 swap을 일으키지 않는 것이 최우선이다.
- 대형 노드는 "상시 로드 모델"과 "필요 시 로드 모델"을 분리한다.
- 이미지 생성과 70B급 추론은 동시에 돌리지 않는 것을 기본값으로 한다.
- 제어 노드가 메모리 압박을 받으면 벡터 DB나 일부 분석 파이프라인을 대형 노드로 승격한다.

### 7.3 영속 저장과 임시 실행의 분리

- 영속 저장: 관계형 DB, 벡터 DB, 에이전트 스킬 DB, 에피소딕 메모리
- 임시 실행: 브라우저 세션, 대형 모델 캐시, 이미지 생성 프로세스

이 분리를 지키면 장애가 나도 **무엇을 복구해야 하는지**가 명확해진다.

### 7.4 데이터 분류 원칙

모든 데이터는 생성 시점에 아래 분류를 따른다.

| 분류 | 저장 위치 | 예시 |
|---|---|---|
| 구조화된 운영 데이터 | 관계형 DB (PostgreSQL) | 작업 이력, 스케줄 정의, 설정값, 메타데이터 |
| 검색용 임베딩 | 벡터 저장소 | 문서 임베딩, 의미 검색 인덱스 |
| 에이전트 기억 | SQLite (Hermes 내장) | 에피소딕 메모리, 스킬 정의 |
| 원본 파일 | 파일 스토리지 | 수집된 문서, 생성된 이미지, 로그 파일 |
| 민감 수집물 | 마스킹 후 관계형 DB 또는 저장 안 함 | 외부 사이트 크롤링 결과 중 개인정보 포함 데이터 |
| 임시 산출물 | 추론 노드 로컬, 작업 완료 후 삭제 | 모델 캐시, 브라우저 세션 파일 |

### 7.5 임베딩 모델 변경 시 재색인 정책

임베딩 모델이 변경되면 기존 벡터와 새 벡터는 호환되지 않는다. 따라서 임베딩 모델 변경은 벡터 저장소 전체 재색인을 수반한다. 원본 데이터는 항상 관계형 DB 또는 파일 스토리지에 보존하여 재색인이 가능하게 한다.

## 8. 네트워크와 보안 철학

### 8.1 Private Network First

모든 내부 서비스는 사설 메쉬 네트워크 기준으로 설계한다.

- 모델 런타임
- 수동 관제 UI
- 벡터 DB
- 이미지 생성 API
- SSH 제어 경로

원칙적으로 공개 인터넷에 직접 노출하지 않는다.

### 8.2 메쉬 네트워크 장애 대비

사설 메쉬 네트워크(Tailscale 등)는 외부 인증 서버에 의존하므로 그 자체가 장애점이 될 수 있다. 다음을 확보한다.

- 두 노드 간 로컬 LAN IP를 통한 직접 연결 경로를 사전 설정
- 메쉬 네트워크 불통 시 로컬 LAN으로 자동 또는 수동 폴백
- 폴백 경로에서도 기본 인증(SSH 키, 토큰 등)은 유지

### 8.3 최소 권한과 최소 노출

- 추론 노드의 LLM API는 제어 노드와 관리자 단말만 접근 가능
- DB는 애플리케이션과 에이전트만 접근 가능
- 브라우저 자동화 결과는 필요한 데이터만 저장하고 원시 HTML 전체를 무조건 보관하지 않음
- 관리 UI는 계정 생성/권한 관리를 최소 범위로 제한

### 8.4 보안보다 편의를 우선하지 않는다

보안과 편의가 충돌하면 기본값은 보안이다.  
특히 인증 없는 내부 API를 외부 공개 포트로 전환하는 방식은 금지한다.

### 8.5 보안 통제 원칙

원칙 수준의 방향 외에, 실제 검증 가능한 통제를 운영 문서에서 정의한다.

- 비밀값(토큰, 키, 웹훅 URL) 로테이션 주기
- 백업 암호화 방식
- 감사 로그 보존 기간
- 관리자 접근 인증 방식(MFA/SSO 권장)
- 로그 내 민감정보 자동 마스킹 기준
- 브라우저 자동화용 계정·세션 보관 정책

## 9. 운영 철학

### 9.1 단순한 구조가 우선

아래 순서로 복잡도를 늘린다.

1. 단일 에이전트 + 스케줄러
2. 역할별 모델 라우팅
3. 벡터 메모리
4. 웹 자동화
5. 이미지 생성
6. 필요 시 복수 에이전트 인스턴스 / 분산 처리

처음부터 모든 기능을 상시 결합하지 않는다.

### 9.2 인간이 끼어들 수 있어야 한다

완전 자동화를 지향하더라도 아래 경로는 유지한다.

- SSH 수동 진입
- 내부 UI 수동 호출
- 로그 확인
- 서비스 재시작
- 모델 강제 언로드 / 교체

### 9.3 워치독과 자기 복구

에이전트 중심 구조에서 에이전트 자체가 크래시 루프에 빠지면 알림조차 보낼 수 없는 데드락이 발생한다. 이를 방지하기 위해 에이전트 프로세스 외부에 최소한의 워치독 메커니즘을 둔다.

- OS 수준 프로세스 관리(launchd/systemd)가 1차 워치독 역할
- 워치독은 에이전트 프로세스 생존 여부만 확인하고, 비정상 시 재시작
- 반복 크래시 감지 시 별도 채널(메일, SMS 등)로 운영자에게 직접 알림

### 9.4 실행 계약(Execution Contract)

모든 워크플로우는 다음 계약을 따른다.

- **멱등성**: 동일 입력에 대해 재실행 시 부작용 없음
- **부분 결과 보존**: 중간 단계 산출물은 단계 완료 시마다 영속화
- **실패 기록**: 모든 실패는 작업 ID, 실패 유형, 타임스탬프, 원인과 함께 구조화된 로그로 기록
- **우선순위**: 경쟁 자원(Node B의 메모리) 사용 시 작업 유형별 우선순위를 명시적으로 정의
- **타임아웃**: 모든 외부 호출과 모델 호출에 타임아웃 설정 필수

## 10. 현재 환경에서 무난한 참조 구현 예시

아래 표는 "현재 이 역할에 무난한 선택"을 정리한 것이다.  
이는 **예시**이지, 구조적 고정값이 아니다.

| 역할 | 현재 무난한 선택 예시 | 교체 조건 |
|---|---|---|
| 에이전트 런타임 | Hermes Agent | 자기 개선 루프 + MCP 양방향 + 메시징을 동등하게 제공하는 대체재 출현 시 |
| 로컬 모델 런타임 | Ollama 계열 | 다른 런타임이 더 나은 병렬성/서빙 효율 제공 시 |
| 수동 관제 UI | Open WebUI 계열 | 더 단순한 내부 UI나 자체 콘솔이 필요할 때 |
| 브라우저 자동화 | Hermes 내장 Camoufox 우선, Playwright MCP 보조 | 특정 시나리오에서 다른 스택이 우위일 때 |
| 코드 작업 | Claude Code + MCP | 다른 코드 에이전트가 더 잘 맞을 때 |
| 관계형 DB | PostgreSQL 기본 | 특정 애플리케이션 호환성 때문에 다른 DB가 필요할 때 |
| 벡터 저장소 | 초기에는 경량 스토어, 확장 시 보안/규모형 스토어 | 데이터 규모, 인증 요구, 다중 사용자 요구가 커질 때 |
| 이미지 워크플로우 | ComfyUI 계열 | 더 적합한 생성 런타임이 있을 때 |
| 외부 장문 처리 | Stable cloud LLM | 로컬 장문 모델이 충분히 좋아지고 비용 이점이 없어질 때 |
| 프로세스 관리 | launchd (macOS) / systemd (Linux) | OS 변경 시 해당 OS 기본 서비스 관리자 사용 |
| 메시징/알림 | Hermes 내장 메시징 게이트웨이 (14개 플랫폼) | 별도 알림 서비스가 필요한 규모로 성장 시 |

## 11. 에이전트가 따라야 할 판단 규칙

### 11.1 역할 우선 규칙
특정 모델명이 문서에 적혀 있어도, 그 이름을 고정 규칙으로 해석하지 않는다.  
반드시 **역할 → 요구 성능 → 현재 가용 모델** 순으로 판단한다.

### 11.2 Control Node 보호 규칙
제어 노드의 안정성이 추론 품질보다 우선이다.  
제어 노드의 메모리 압박이 커지면 다음 순서로 조정한다.

1. 벡터 스토어 축소 또는 승격
2. 부가 서비스 중지
3. 외부 API 활용 증가
4. 마지막으로 구조 변경

### 11.3 Compute Node 활용 규칙
추론 노드는 적극적으로 활용하되, 아래를 지킨다.

- 상시 로드 모델은 최소화한다.
- 무거운 분석 모델은 필요 시만 로드한다.
- 이미지 생성과 대형 추론은 경쟁 자원으로 본다.
- 브라우저 자동화는 모델보다 먼저 실패할 수 있음을 가정한다.

### 11.4 인터페이스 우선 규칙
가능하면 구성 요소끼리는 아래 접점으로 연결한다.

- HTTP/JSON API
- MCP
- SSH
- DB connection string
- Webhook

이 규칙을 지키면 교체와 디버깅이 쉬워진다.

### 11.5 학습 자산 보존 규칙
에이전트의 스킬 DB와 에피소딕 메모리는 모델이나 런타임 교체 시에도 보존되어야 한다. 이 데이터는 DB 백업과 동등한 우선순위로 관리한다.

## 12. 비목표(Non-goals)

이 인프라는 기본적으로 아래를 목표로 하지 않는다.

- 대규모 멀티테넌트 SaaS
- 공개 인터넷에 직접 노출된 AI 플랫폼
- Kubernetes 수준의 복잡한 분산 스케줄링
- 고가용성(HA) 클러스터
- 모든 기능의 24/7 상시 활성화

필요하면 나중에 확장할 수 있지만, 현재 목표는 **작고 강한 개인/소규모 운영 인프라**다.

## 13. 한 줄 요약

이 아키텍처는 **작은 노드가 시스템을 기억하고 통제하며, 큰 노드가 필요할 때만 무거운 계산을 수행하는 구조**다.  
에이전트는 작업을 수행할수록 **스킬과 기억을 축적**하며, 모델과 도구는 이름이 아니라 **역할**로 선택하고, 시스템은 **프라이빗 네트워크·온디맨드 실행·단순한 운영**을 기준으로 유지한다.
